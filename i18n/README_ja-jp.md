## エージェントはコードを書ける。では、システムを設計できるか？

多くの人は、コーディングエージェントにアーキテクチャ判断を求めません。

求めるのは、普通の開発作業です。

```text
請求書ダウンロード用のエンドポイントを追加して。
二つ目の決済プロバイダーを接続して。
このサービスをリファクタリングして。
この機能を拡張しやすくして。
```

しかし、最初の 1 行が書かれる前に、エージェントはすでにアーキテクチャ上の選択をしています。

- どのモジュールが振る舞いと状態を持つのか。
- どの依存関係が境界をまたぐのか。
- どのエラーが公開された振る舞いになるのか。
- 何を互換のまま保ち、何を意図的に変更してよいのか。
- 直接実装、抽象化、フレームワーク拡張のどれに根拠があるのか。

アーキテクチャ判断がないと、エージェントは高くつく二つの方向のどちらかで失敗しがちです。

| 失敗 | コードベースがどうなるか |
| --- | --- |
| **放置されたアーキテクチャ** | 機能は最寄りのファイルに追加されます。責務は曖昧になり、ルールは重複し、分岐は増え、いくつかのサービスやパッケージは誰も触りたがらない大きさまで膨らみます。 |
| **憶測によるアーキテクチャ** | 現実の変化が存在する前に、あり得る未来がインターフェース、ファクトリー、レジストリ、イベント、ラッパー、継承階層へと変わってしまいます。 |

そして、この二つの中に隠れている三つ目の失敗があります。

> **思い込みによる互換性。** エージェントは、境界を明確にしないまま、古い振る舞いを当然のように残すか、動いている振る舞いを書き換えてしまいます。

結果はコンパイルできるかもしれません。デモも通るかもしれません。

それでも、次の変更が来るたびに、理解しづらく、テストしづらく、拡張しづらく、信頼しづらいコードベースになります。

**Agent Architect は、複雑さが広がる前に構造を考えるための規律ある方法を、コーディングエージェントに与えます。**

## 導入前と導入後

### Agent Architect なし

```text
ユーザー: 「二つ目の決済プロバイダーを追加して」

エージェント: CheckoutService にプロバイダー固有の条件分岐を追加し、
Webhook 処理を複製し、ハンドラから永続化層に直接触れ、
「将来のプロバイダーのため」に PaymentProviderFactory を作り、
既存連携が使っているリトライ挙動を静かに変更する。
```

### Agent Architect あり

```text
ユーザー: 「二つ目の決済プロバイダーを追加して」

エージェント:
1. 既存の決済ライフサイクル、呼び出し元、リトライ責務、
   Webhook、永続化、エラーパス、テストを整理する。
2. 古い契約を維持すべきだと決めつけたり、
   逆に壊してよいと決めつけたりせず、互換性の意図を明確にする。
3. 実証済みの変化点（プロバイダー実行）と、
   安定した方針（チェックアウト、認可、リトライ責務）を分離する。
4. 最小の直接設計と、プロバイダー境界を切る設計を比較する。
5. 証拠のある変化を隔離するために必要な構造だけを導入する。
6. 合意した互換性境界の中で、成功、拒否、タイムアウト、
   Webhook、リトライ、移行、ロールバック経路を検証する。
```

結果は、儀式が増えることではありません。

それは **影響範囲が小さくなり、設計が長持ちし、その判断をコードベースが代償を払う前に確認できるようになること** です。

## Agent Architect が変えること

Agent Architect は、アーキテクチャをユーザー向けの委員会作業に変えるものでは **ありません**。

高速なコード生成器を、より責任あるソフトウェア設計者に変え、判断が振る舞い・構造・互換性に影響する場面で、その思考を見えるようにします。

| 代わりに起きがちなこと | Agent Architect がエージェントを助けること |
| --- | --- |
| いちばん近いファイルにコードを書く | 本当の責務の持ち主、境界、呼び出し元、依存関係、失敗経路を見つける。 |
| 起こり得る未来ごとに抽象化する | 独立した変化が名前付きで証拠化されるまで、直接設計を保つ。 |
| 後方互換か全面書き換えかを思い込みで決める | 設計前に、意図する互換性境界を明確にする。 |
| リファクタリングを「きれい」とだけ呼ぶ | 代替案、構造コスト、行った検証、残るリスクを明示する。 |

> 互換性の意図は官僚主義ではありません。誰も必要としていないレガシー挙動を守り続けることと、誰かが依存している挙動を壊すことという、同じくらい高くつく二つの失敗を防ぎます。

## どうやってより良いアーキテクチャを作るか

### 1. 設計の前に現実を読む

エージェントは構造を提案する前に、リポジトリ上の証拠を確認します。影響を受ける呼び出し元、責務、依存関係、状態、失敗経路、ライフサイクル、トランザクション、並行性、フレームワーク制約、既存テストを調べます。

### 2. 互換性を意図的に扱う

変更が契約、データ、設定、外部連携、拡張ポイントに影響し得るなら、何を互換のまま保ち、何を意図的に変えてよいかを確認します。そのうえで、実際の境界、移行やロールバックへの影響、未解決のリスクを記録します。

「全部そのまま守る」や「きれいに書き直す」を黙って選ぶことはしません。

### 3. いちばん小さく長持ちする構造を選ぶ

直接解は、常に現実的な代替案です。インターフェース、アダプター、ストラテジー、イベント、ファクトリー、ラッパー、フレームワーク拡張は、具体的で独立した変化を隔離できるときにだけ、そのコストを正当化できます。

### 4. 判断を説明し、検証する

エージェントは監査可能なアーキテクチャ記録を作り、影響を受ける境界で振る舞いを検証します。何を実行したか、何がまだ不確かか、却下した代替案がなぜコストに見合わなかったかを報告します。

## エージェントが出力するもの

Agent Architect は、単純ではない機能追加、連携、設計、リファクタリング、レビューごとに、アーキテクチャ記録を生成します。

```text
01. 設計診断
    目的、非目標、リポジトリ上の証拠、呼び出し元、安定した核、
    変化点、失敗モード、制約。

02. 互換性の意図
    維持する契約と意図的に変更する契約、利用者、
    移行またはロールバック境界、未解決リスク。

03. 代替案と判断
    最小の直接設計、正当化された構造、却下した選択肢、
    API への影響、依存方向、継続コスト。

04. 検証
    正常系、境界、失敗、統合、並行性、運用面の確認、
    実際に行った検証、残る不確実性。
```

こうして、アーキテクチャ判断は説明可能になります。今日たまたま動く、生成されたファイルの山ではなくなります。

## 使い方

コーディングエージェントに次のように依頼します。

```text
この非自明な変更を実装またはレビューする前に $agent-architect を使ってください。
リポジトリを調べ、必要な箇所では互換性の意図を明確にし、
最小で長持ちするアーキテクチャを選んでから、実装と検証を行ってください。
```

例:

```text
この決済プロバイダーを追加する前に $agent-architect を使ってください。

この機能に偶然のアーキテクチャや、
憶測による抽象化、曖昧な互換性境界がないか $agent-architect でレビューしてください。

このサービス境界をリファクタリングする前に $agent-architect を使ってください。
契約を明確にしたうえでのみ、振る舞いを維持または意図的に変更してください。

この連携に必要なのがアダプターか、
直接依存か、フレームワーク拡張かを $agent-architect に判断させてください。
```

## 使うべき場面

変更がコードベースの形を変え得るなら、Agent Architect を使ってください。

- 機能がモジュール、レイヤー、サービス、データベース、外部プロバイダーをまたぐ。
- 「リファクタリングする」「拡張可能にする」「抽象化する」「疎結合にする」「一般化する」「将来に備える」をやろうとしている。
- エージェントがインターフェース、ファクトリー、イベント、レジストリ、ラッパー、継承階層、グローバル状態を提案している。
- 既存の振る舞いを互換のまま保つべきか分からない。
- 巨大サービス、巨大パッケージ、重複ルール、レイヤー横断依存、誰も自信を持って説明できない分岐ロジックが見えている。
- PR は一見動いているが、構造上の判断がまだ暗黙のままになっている。

> [!TIP]
> Agent Architect は、機能開発と構造変更を扱うワークフローに組み込んでください。偶発的な複雑さが広がってからでは遅いです。

## パターンは出発点ではなく、最後の一手

Agent Architect は GoF 23 パターンをすべて扱えます。

しかし、もっと重要なのは **いつ使わないかを判断できること** です。

出発点は証拠です。**何が変わるのか、誰が責務を持つのか、何が壊れ得るのか、何を安定させる必要があるのか、そして直接解がすでに何を解決しているのか。**

<details>
<summary><b>生成に関する判断</b></summary>

- Factory Method
- Abstract Factory
- Builder
- Prototype
- Singleton scope

</details>

<details>
<summary><b>構造に関する判断</b></summary>

- Adapter
- Bridge
- Composite
- Decorator
- Facade
- Flyweight
- Proxy

</details>

<details>
<summary><b>振る舞いに関する判断</b></summary>

- Chain of Responsibility
- Command
- Interpreter
- Iterator
- Mediator
- Memento
- Observer
- State
- Strategy
- Template Method
- Visitor

</details>

<details>
<summary><b>重要なパターン境界</b></summary>

| 混同してはいけないもの | 区別すべき相手 |
| --- | --- |
| Decorator | Proxy または Adapter |
| Facade | Mediator |
| Factory Method | Abstract Factory、Builder、Prototype |
| Strategy | State または Template Method |
| Observer | Chain of Responsibility または Command |
| Composite | Decorator |

パターンは **意図、協調者、ライフサイクル、変化、失敗時の振る舞い** によって選びます。クラス図だけで選んではいけません。

</details>

## このスキルは何ではないか

| これは違う | これはこういうもの |
| --- | --- |
| パターン百科事典 | コーディングエージェント向けのアーキテクチャ判断システム |
| 儀式を増やす装置 | 適切な量の構造を明示し、検証可能にする方法 |
| デフォルトで「クリーンアーキテクチャ」 | 境界、責務、依存関係、ライフサイクルを証拠ベースで分析すること |
| 層を増やす理由 | 証拠がないときに直接設計を維持してよいという許可 |
| 人の承認を詰まらせるボトルネック | 互換性やアーキテクチャが重要になったときに人へ見える形で伝える方法 |
| エンジニアリング責務の代用品 | エージェントが生み出す構造を、より責任ある、監査可能で、保守しやすいものにする方法 |

## コントリビュート

コントリビューションは **アーキテクチャ判断を良くすること** に寄与すべきであり、儀式を増やすべきではありません。

有益な貢献には、次のようなものがあります。

- 実際の機能追加やアーキテクチャ判断に対する証拠ゲート。
- より明確な互換性境界、移行ガイダンス、ロールバック基準。
- より明確なパターン境界と却下基準。
- 偶然のアーキテクチャや憶測による抽象化を再現できる例。
- ライフサイクル、並行性、トランザクション、移行、ロールバックに対する検証ガイダンス。
- エージェントが構造的複雑さを広げにくくなる修正。

変更を出す前に、次を自問してください。

```text
これは、コードが広がる前にエージェントが下せる判断を改善するか？
この改善は検証できるか？
憶測的なプロセスを増やさずにガイダンスを足しているか？
```
